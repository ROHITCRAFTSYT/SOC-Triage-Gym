01 / THREAT BRIEF

Why this exists

THE PROBLEM

A real Security Operations Center drowns in alerts. Analysts triage thousands of signals a day; most are noise, a few are live intrusions, and the cost of getting it wrong is asymmetric in both directions — miss a true positive and you're breached, isolate a healthy host and you've taken down production.

LLM agents are an obvious fit for triage — but almost every RL environment trains a single agent in isolation. Real SOCs don't work that way. They have tiers, escalation paths, audit oversight, ticketing SLAs, policies that change mid-shift, and users who email in "this looks weird" at the worst possible moment.

If you want agents that operate inside a real security workflow, you have to train them inside one — team structure, ticket bus, drifting policies and all. // DESIGN THESIS

SOC-Triage-Gym models the full workflow: Tier-1 triages and escalates, Tier-2 contains confirmed threats, a Manager audits the team — with a reward signal that punishes any agent maximizing personal score at the team's expense.

  • [✓]
    Team, not soloThree cooperating roles with a live ticket bus and a phase state machine — the reward blends individual performance with team F1.
  • [✓]
    Verifiable rewards (RLVR)6+ layered programmatic graders. No LLM-only scoring — every reward component is a deterministic, testable check.
  • [✓]
    Adaptive environment (RLVE)A Red-Team Generator co-evolves scenario difficulty with the agent, holding the blue-team win rate near 50% — the learning zone.
  • [✓]
    Hostile to reward hackingSix named exploit vectors were found, fixed, and locked in as regression tests. NOOP-spam, duplicate closes, escalation flooding — all patched.
  • [✓]
    Reproducible by designSame seed → same episode → same score. Deterministic replay is what makes per-step GRPO training possible at all.

CONTEXT SHIFTDRIFT

Policy Drift

The rulebook changes mid-episode — a field gets renamed, a severity threshold moves, admin accounts suddenly require escalation. Graders honour the policy version active at action time, so rules that changed at step 30 can't retroactively re-score step 1.

Rotating Experts

A rotating expert panel — Dr. Accuracy, Speedy Sam, Thorough Thea — scores episodes with drifting rubric weights. The agent is hinted which expert is judging and must adapt its style: precision, speed, or depth.

05 / SIGNAL DESIGN

Reward & Integrity

RLVR — VERIFIABLE BY CONSTRUCTION

Dense step rewards pay for productive investigation; the final score on submit is the grader score (0–1) scaled by an efficiency multiplier. Solve early and keep everything; drag past 90% of the step budget and the score is taxed to ×0.70. A reward function is only as good as the exploits it can't be farmed by — six exploit vectors were found, patched, and locked in as regression tests.

06 / LEARNING LOOP

GRPO Training

PER-STEP · NOT TRAJECTORY-AVERAGED

Training uses real per-step GRPO. Each dataset row is a single decision point; the reward function replays the environment deterministically to that step (same seed → same state), applies the model's one action, and returns the environment's immediate blended reward. No full-episode oracle rollouts, no averaged trajectories. 

# one dataset row = one decision point
row    = (observation@step_index, task_id, seed, step_index)

# reward = replay env to step_index, apply model action
env.reset(task_id, seed)
replay_to(step_index)              # deterministic
reward = env.step(model_action).reward   ← immediate signal

# train on Colab / Kaggle T4
$ python train_grpo.py --role tier1 \
    --model Qwen/Qwen2.5-0.5B --unsloth

# dry-run oracle baseline, no GPU needed
$ python train_grpo.py --role tier1 --dry-run

Stack: Unsloth 4-bit QLoRA · TRL GRPO · Qwen2.5-0.5B. The trained tier-1 adapter is published at huggingface.co/rohitcraftsyt/soc-grpo-tier1 and the full walkthrough is a one-click Colab notebook.

RANDOM POLICY — UNTRAINED FLOOR0.063

The environment bounds the problem on both sides: a uniform-random policy is the floor an untrained LLM competes against; the scripted oracle is the ceiling a trained one should approach. Both curves are committed to the repo and regenerate deterministically.

Honest limitations

LIM-01

Only tier-1 is RL-trained end-to-end. Tier-2 and Manager are played by the scripted oracle during tier-1 GRPO — a staged curriculum, by design.

LIM-02

The published run underfit. 27 optimizer steps on a free T4 wasn't enough to move a small model off its baseline. The loss decreases, the checkpoint reloads, the eval is deterministic — the pipeline is ready; the compute wasn't. A10G projects ~5h for the full config.

LIM-03

The manager-judge fallback is a keyword heuristic. The LLM path is primary; the fallback is bounded and tested, but not production-grade.

LIM-04

The narrative grader is length-sensitive, not semantics-sensitive. Token caps limit abuse, but a semantic LLM judge is future work.

// WHY IT FITS THIS MEETUP

Memory

Episode state, the ticket bus, and /policy/history are explicit, queryable memory. Nothing lives in a prompt — an agent that forgets what Tier-1 escalated, or which policy version was active at step 12, pays for it in reward.

// WHY IT FITS THIS MEETUP

Context

Context isn't static here — it drifts. Mid-episode policy changes, unsolicited NPC messages, and 250-step horizons force agents to manage context windows the way real analysts manage a shift: triage what matters, discard what doesn't.

// WHY IT FITS THIS MEETUP

Agents

Not one agent — a team of three with an adversary underneath. Coordination is rewarded, selfishness is penalized, and the red-team generator is itself an agent learning to break the defenders.

The demo script hits five beats in sequence — the same arc as a live presentation: establish the baseline, open up the grader, show the trained policy, measure the delta, and prove the safeguards are on.

Everything is deterministic: same seed, same episode, same score — so the demo you rehearse is the demo you give.

Run the scripted oracle on the phishing task — establishes the ceiling behaviour.

python inference.py --task phishing --seed 42

curl -X POST localhost:7860/grader -d '{"task_id":"phishing"}'
